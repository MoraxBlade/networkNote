### 2.1.2进程通信（processes communicating）

#### 进程
（Process）：在主机上运行的应用程序

- 在同一个主机内，使用进程间通信机制 IPC 进行通信（操作系统定义）
    
- 不同主机，基于应用协议，通过交换报文（message）来通信
    

#### 客户和服务器进程

**客户**：发起通信的进程

**服务器**：等待连接的进程

#### 套接字（socket，进程和计算机网络间的接口）

**核心概念**：是应用层进程与运输层之间的软件接口，是应用程序和网络之间的应用编程接口（API），是进程向网络发送报文、从网络接收报文的唯一入口

**层级与控制权**：

- 套接字位于同一台主机的应用层（进程）与运输层之间，是应用程序与网络的边界接口
    
- **应用程序开发者**：可控制套接字的应用层端，仅能对运输层进行有限控制
    
    - 选择运输层协议
        
    - 设定部分运输层参数（如最大缓存、最大报文段长度等）
        
    - 无法控制套接字的运输层端与运输层的具体传输逻辑
        
- **操作系统**：负责控制套接字的运输层端，以及运输层协议（如TCP）的缓存、变量等底层实现
    

**工作流程**：

- 发送端：应用进程将报文写入套接字，由操作系统控制的运输层（如TCP）处理后，经因特网传输
    
- 接收端：因特网传输的报文抵达后，由操作系统控制的运输层处理，再从套接字交付给应用进程，由进程完成报文处理
    

#### 进程寻址

用于标识收的进程，需要两种信息

- 主机的地址（IP，32位）
    
- 在目的主机中指定接收该进程的标识符
### 2.1.3应用层协议——如何交换报文

#### 定义

定义了运行在不同端系统上的应用程序进程之间，相互传递报文的规则，是网络应用的核心组成部分（但不等同于整个网络应用）

#### 要素

- **交换的信息类型**：如请求（request）、响应（response）
    
- **消息的语法（syntax）**：格式结构，消息里包含哪些字段、这些字段用什么格式来描述（比如二进制、ASCII）
    
- **消息语义（semantics）**：每个字段里信息的具体含义，比如某个字段代表状态码、某个字段代表请求路径
    
- **进程何时以及如何发送和响应消息的规则**：通信的时序逻辑，比如什么时候发请求、收到请求后多久要响应、异常情况怎么处理，相当于通信的“对话流程”
### 2.1.4 需要的运输服务

- **可靠数据传输**
- **吞吐量**
  - 带宽敏感的应用：具有吞吐量要求；
  - 弹性应用：根据需要充分利用可够使用的吞吐量。
- **定时保证**
- **安全性**
### 2.1.5 因特网提供的运输服务

#### TCP service

**TCP服务特性**

- **可靠的传输服务**：无差错、按序、无丢失无冗余地交付所有发送数据，保障数据完整性
    
- **面向连接**：要求在客户端进程和服务器进程之间建立连接

- **流量控制**：匹配收发双方速率，避免发送方数据淹没接收方
    
- **拥塞控制**：网络拥塞时抑制发送方发送速率，公平共享网络带宽，保障网络整体稳定
    
- **不提供的服务**：时间保证、最小吞吐保证、原生安全加密（TCP/UDP均为明文传输）
    

#### UDP协议

**UDP服务特性**

- **不可靠数据传输**：不保证数据无差错、按序、完整交付，报文可能乱序、丢失
    
- **不提供的服务**：可靠传输、流量控制、拥塞控制、时间保证、带宽保证、建立连接
    
- **核心优势（UDP存在的必要性）**：
    
    - 能够区分主机上的不同应用进程            
    - 无需建立连接，无握手开销，延迟极低，适合事务型、低延迟需求的应用
    - 时间开销小，适合对实时性要求高、对正确性要求不高的场景（如音视频通话）
    - 无拥塞/流量控制，应用可按设定速率发送数据，不受TCP速率限制
        - 而在TCP上面的应用，应用发送数据的速度和主机向网络发送的实际速度 是不一致的
#### **TCP安全增强：TLS（传输层安全）**
**原生问题**：TCP/UDP套接字均无加密，明文传输
**TLS核心特性**：
- 在TCP之上、应用层实现，是TCP的加强版，不改变TCP基础传输逻辑
- 提供三大**核心安全能力**
    - 数据加密
        
    - 数据完整性校验
        
    - 端到端端点鉴别
        
- 有独立的套接字API，应用程序调用TLS套接字，明文经TLS加密后再交给TCP传输，接收端TLS解密后交付应用
#### 运输层协议不提供的服务

- 不提供吞吐量保证、定时保证
    
- 原生无加密

## 2.2 Web应用和HTTP协议

### 2.2.1 HTTP概况

#### 基本概念

**HTTP**：超文本传输协议，hypertext　transfer　protocol

**作用**：Web的应用层协议
    
**模式**：C/S模式

#### TCP——HTTP的支撑运输协议

1. 客户端发起一个与服务器的TCP连接(建立套接字)， 服务器端口号为 80
    
2. 服务器接受客户端的TCP连接请求
    
3. 浏览器(HTTP客户端) 与 Web服务器(HTTP服务器)交换HTTP 报文 (应用层协议报文)
    
4. TCP连接关闭
    
#### 无状态

- HTTP 是无状态的（stateless）
    
- 服务器并不维护关于客户端的任何信息
    

**原因**：维护状态的协议很复杂！

- 必须维护历史信息 (状态)
    
- 如果服务器/客户端死机，它们的状态信息可能不一致 （但二者的信息需要保持一致）
    
- 无状态的服务器能够支持更多的客户端

### 2.2.2 非持久连接和持久连接
#### RTT

RTT（Round-Trip Time）：一个小分组从客户机到服务器再回到客户机所花费的时间。

#### 非持久连接
- 每个请求及相应的响应对应一个单独的TCP连接发送
- **缺陷**：每个对象消耗2个RTT，操作系统需频繁为TCP连接分配资源，效率低下
#### 持久连接
- 所有请求及相应的响应经相同TCP连接发送。服务器在发送响应后保持该TCP连接打开
- **方式（流水\非流水）**
    - **非流水方式**：客户端需收到前一个响应后才能发送下一个请求，每个对象仍需消耗约1个RTT
    - **流水方式（默认模式）**：客户端遇到引用对象立即生成请求，无需等待前一个响应，所有引用对象理论上仅消耗1个RTT，效率大幅提升
- **优势**：减少TCP连接建立/拆除开销，降低服务器资源消耗，提升传输速度

### 2.2.3 HTTP报文格式

HTTP报文有两种：请求报文、响应报文。

#### 请求报文
**报文结构**

**1. 请求行**
- **方法**：GET、POST、HEAD、PUT、DELETE
- **URL**：请求资源路径
- **HTTP版本**：如 HTTP/1.1
- 示例：`GET /index.html HTTP/1.1`

**2. 首部行**
- 格式：字段名: 值
- 常见字段：
  - **Host**：目标主机
  - **Connection**：连接策略（持久/关闭）
  - **User-agent**：客户端标识
  - **Accept**：可接收的内容类型
  - **Accept-encoding**：支持的压缩编码（如gzip）
  - **Accept-language**：偏好的自然语言

**3. 实体体（Entity Body）**
- 承载实际数据（如POST提交内容）
- GET方法通常为空

**常用方法对比**

| 方法       | 作用           | 数据位置           | 特点                         |
| :--------- | :------------- | :----------------- | :--------------------------- |
| **GET**    | 获取资源       | URL中（`?`后传参） | 长度受限，安全性低，无实体体 |
| **POST**   | 提交数据       | 实体体中           | 适合大体积数据，更安全       |
| **HEAD**   | 仅获取报文首部 | 无                 | 不返回对象，用于故障跟踪     |
| **PUT**    | 上传/替换文件  | 实体体中           | 将内容写入指定URL            |
| **DELETE** | 删除指定资源   | 无                 | 请求服务器删除URL指向的文件  |

#### 响应报文

**报文结构**
- **状态行**：协议版本、状态码、短语
- **首部行**：格式：字段名: 值
  - 包含服务器信息、资源属性和缓存控制字段
  - **Date**：响应生成时间
  - **Server**：服务器软件版本
  - **Last-Modified**：资源最后修改时间
  - **Content-Length**：实体体字节数
  - **Content-Type**：实体体数据类型（如 text/html）
  - **ETag**：资源唯一标识（缓存校验）
  - **Connection**：连接控制（close/keep-alive）
- **实体体**：请求的目标资源（HTML、图片、JSON等）


**常用状态码**

| 类别               | 状态码 | 短语                       | 含义                              |
| :----------------- | :----- | :------------------------- | :-------------------------------- |
| **2xx 成功**       | 200    | OK                         | 请求成功，对象在实体体中          |
| **3xx 重定向**     | 301    | Moved Permanently          | 资源永久转移，新URL在Location字段 |
|                    | 304    | Not Modified               | 资源未修改，可使用缓存            |
| **4xx 客户端错误** | 400    | Bad Request                | 请求语法错误                      |
|                    | 404    | Not Found                  | 服务器无此资源                    |
| **5xx 服务器错误** | 505    | HTTP Version Not Supported | 不支持的HTTP版本                  |
### 2.2.3 用户与服务器的交互：cookie
#### 核心作用
- 为HTTP补充**状态管理能力**（HTTP本身无状态）
- 实现：身份识别、购物车、个性化推荐、会话维持


#### 四大组成部分
| 组成部分         | 说明                                               |
| :--------------- | :------------------------------------------------- |
| **响应首部**     | 服务器通过 `Set-cookie` 首部下发唯一识别码给客户端 |
| **请求首部**     | 浏览器后续请求通过 `Cookie` 首部回传识别码         |
| **客户端存储**   | 浏览器在本地管理 cookie 文件                       |
| **服务器端存储** | 网站后端数据库维护 cookie 对应的用户数据           |


#### 工作流程

| 阶段           | 过程                                                           |
| :------------- | :------------------------------------------------------------- |
| **首次访问**   | 服务器生成唯一ID → 后端建表 → 响应带 `Set-cookie` → 浏览器存储 |
| **后续访问**   | 请求自动带 `Cookie: ID` → 服务器识别用户 → 执行个性化操作      |
| **跨时段访问** | 本地 cookie 持久化 → 服务器可恢复用户状态                      |


#### 应用场景
- 用户验证（登录身份确认）
- 购物车（关联用户商品）
- 个性化推荐（基于历史行为）
- 会话维持（如Web邮箱）


#### 隐私争议
- 允许站点获取用户信息，存在隐私隐患
- **第三方Cookie**：跨站点跟踪用户，侵犯用户隐私

### 2.2.4 Web缓存


#### 定义
- 能代表初始Web服务器满足HTTP请求的网络实体
- 拥有本地磁盘存储空间，保存最近请求过的对象副本
- **既是服务器又是客户端**：接收浏览器请求并发回响应；向初始服务器发起请求


#### 工作流程
1. 浏览器将所有HTTP请求发送到Web缓存
2. **缓存命中**：缓存直接返回本地副本
3. **缓存未命中**：缓存向原始服务器请求，保存副本后再返回给客户端


#### 使用原因
- **降低响应时间**：缓存更接近客户端，瓶颈带宽更高
- **减少接入链路流量**：可以大大减少一个机构内部网络与 Internet接入链路上的流量
  - 降低机构带宽需求和费用
- **提升内容交付能力**：使较弱的内容提供商也能高效交付


#### 缓存效果对比（瓶颈分析）

| 场景                      | 接入链路带宽 | 接入链路利用率 | 总端到端延时                        |
| :------------------------ | :----------- | :------------- | :---------------------------------- |
| **无缓存（基准）**        | 1.54 Mbps    | ≈ 97%          | ≈ 4.1s（排队≈2.1s + RTT 2s）        |
| **升级链路**              | 154 Mbps     | ≈ 0.97%        | ≈ 2s（仅消除排队，RTT不变）         |
| **部署缓存（命中率40%）** | 1.54 Mbps    | ≈ 58%          | ≈ 1.2s（命中请求≈0s，未命中≈2.01s） |

- **核心结论**：部署缓存成本低（仅需一台服务器），效果优于升级昂贵链路


#### 缓存更新：条件GET方法

**目的**：验证缓存副本是否为最新，避免不必要的数据传输

**请求格式**：`GET` + `If-Modified-Since: <date>`

**判定逻辑**：
| 服务器判断 | 响应                                                        |
| :--------- | :---------------------------------------------------------- |
| 资源已更新 | 返回新对象（200 OK）                                        |
| 资源未更新 | 返回 **304 Not Modified**（不含实体体），告知缓存可使用副本 |

## 2.3 因特网中的电子邮件

#### 组成部分：
- **用户代理**：邮件阅读器，用于撰写、编辑和阅读邮件，邮件保存在服务器上。
- **邮件服务器**：管理用户邮箱（接收）和报文队列（待发送）。
- **SMTP**（简单邮件传输协议）：主要应用层协议，使用TCP可靠传输，发送方服务器为客户端，接收方服务器为服务端。
### 2.3.1 SMTP

#### SMTP 定义与基础

依据标准：RFC 5321 定义了 SMTP，它是因特网电子邮件的核心

**传输协议**：使用 TCP 在客户端和服务器之间传送报文，端口号为 25

**传输方式**：支持直接传输，即从发送方服务器到接收方服务器

SMTP不使用中间邮件服务器发送邮件（direct transfer），即使这两个服务器位于地球两端。SMTP在25号端口创建TCP连接。

#### 传输阶段：
1. **握手阶段**：SMTP客户机指明发送方和接收方邮件地址
2. **发送报文**：利用TCP连接提供的可靠数据传输将报文从发送邮件服务器传输到接收服务器
3. **关闭连接**

#### SMTP 命令与响应与编码

- 交互形式：采用命令/响应交互，类似 HTTP，均为 ASCII 文本形式。
    
    - 命令 (commands)：由客户端发送，为 ASCII 文本
        
    - 响应 (response)：由服务器返回，包含状态码和状态信息
        
- 报文**编码**：报文必须为 7位 ASCII 码，取值范围 0-127

#### 简单的SMTP报文交互

- 客户主机名：crepes.fr
    
- 服务器主机名：hamburger.edu
    

```
S: 220 hamburger.edu
C: HELO crepes.fr
S: 250  Hello crepes.fr, pleased to meet you
C: MAIL FROM: <alice@crepes.fr>
S: 250 alice@crepes.fr... Sender ok
C: RCPT TO: <bob@hamburger.edu>
S: 250 bob@hamburger.edu ... Recipient ok
C: DATA
S: 354 Enter mail, end with "." on a line by itself
C: Do you like ketchup?
C: How about pickles?
C: .
S: 250 Message accepted for delivery
C: QUIT
S: 221 hamburger.edu closing connection
```
SMTP客户机与服务器之间交换报文脚本的例子：
- HELO是HELLO的缩写；
- HELO、MAIL FROM、RCPT TO、DATA、QUIT等命令均是自解释的；
- 客户机通过发送一个只含一个句点“.”的行，告诉服务器报文结束。

与SMTP服务器直接对话：telnet ServerName 25

### 2.3.2 与HTTP的对比

两者的重要区别：
1. HTTP是一个拉协议，；SMTP是一个推协议
2. SMTP要求每个报文都使用7位ASCII格式，HTTP没有这个限制；
3. HTTP把每个对象封装到它自己的HTTP响应报文中，而因特网电子邮件把所有报文对象放在一个报文之中

### 2.3.4 邮件访问协议

#### 基本概念
- **目的**：访问服务器取回邮件
- **方法**：
  - HTTP：基于Web的电子邮件或智能手机客户端
    - 在SMTP、IMAP/POP3基础上提供Web界面
  - POP3：邮局访问协议 [RFC 1939]
  - IMAP：Internet邮件访问协议 [RFC 3501]

#### POP3
- 两个阶段
    - 用户确认阶段
        
    - 事物处理阶段
        
- 两种配置模式
    
    - 下载并删除（节省空间）
        
    - 下载并保留
- 会话过程中无状态

#### IMAP
- 服务器将每个邮件与文件夹关联，支持目录组织报文和远程管理文件夹
- 允许读取报文部分内容，适用于低带宽连接
- 会话过程中保留用户状态（目录名、报文ID与目录名映射）

## 2.4 DNS：因特网的目录服务

### 2.4.1 DNS提供的服务

#### 主机名与IP地址
DNS提供主机名到IP地址的转换服务
- 主机名便于人类记忆
- IP地址便于路由器处理。
#### 定义
DNS是由分层DNS服务器实现的分布式数据库，同时提供查询协议；该协议运行于UDP之上，端口号为53。
#### 主要服务：
  - 主机名到IP地址的转换（核心功能）
  - 主机别名：将别名映射到规范主机名
  - 邮件服务器别名：为邮件服务器提供别名解析
  - 负载分配：在冗余服务器间循环返回IP地址，实现负载均衡
#### 使用流程
- 应用程序（如浏览器）提取主机名
- 通过DNS客户端查询DNS服务器
- 获取对应的IP地址后，再建立连接（如TCP）

### 2.4.2 DNS工作机理概述

#### 集中式DNS服务器的问题：
1. 单点故障：如果该DNS服务器崩溃，整个因特网将随之瘫痪；
2. 通信容量：单个DNS服务器不得不处理所有的DNS查询；
3. 远距离的集中式数据库：单个DNS服务器不可能临近所有查询客户机，远距离查询可能导致严重时延；
4. 维护：DNS中央数据库十分庞大，且需为不断添加的主机而频繁更新。

#### 分布式、分层的数据库结构
- **根域名服务器**：由ICANN管理，全球13个逻辑根（每个备份多次），负责顶级域指引
- **顶级域名服务器（TLD）**：负责通用顶级域（如.com、.org）和国家顶级域（如.cn、.uk）
- **权威域名服务器**：组织机构维护，提供本域内主机名到IP的权威映射
- **本地DNS服务器**：不属于层次结构，由ISP提供，缓存查询结果，充当主机的第一查询点


#### DNS命名空间
- 域名用点分隔各级标签（如`sysu.edu.cn`）
- 域可包含子域（如`sse.sysu.edu.cn`是`sysu.edu.cn`的子域）
- 主机域名唯一标识一台主机
- 域的管理权由父域授予子域，域与物理网络无关
#### 域名服务器与区域
- 区域是命名空间的互不相交划分，由区域管理者决定（树结构）
- 域名服务器分为：根、TLD、权威、本地（后三者按需配置）
#### DNS解析过程
- 客户端调用解析器，向本地DNS服务器发起查询
- 若本地无法解析，可采用两种查询方式：
  - **递归查询**：本地DNS逐级向上级服务器转发查询，最终将结果返回给客户端，各级服务器承担全部查询负载（根压力大）
  - **迭代查询**：本地DNS依次询问各层级服务器，每层返回下一级服务器的地址，最终由权威服务器返回IP，根服务器仅返回TLD地址


#### DNS缓存
- 服务器缓存查询得到的映射，提高响应速度
- 缓存条目设置TTL（默认两天），超时后删除
- 缓存可能过期，导致映射不准确（尽力而为的转换）

### 2.4.3 DNS记录和报文

#### 资源记录⭐
- **作用与位置**：维护域名-IP等映射关系，存储于DNS分布式数据库中
- **格式**：`Name, Value, Type, TTL`
  - TTL为生存时间（权威/缓存记录）
- **主要Type类型**：
  - **A**：标准主机名→IP映射（Name=主机名，Value=IP）IPv4
  - **AAAA**：主机名→IPv6
  - **NS**：指向域名的权威服务器，用于沿着查询链路进一步路由DNS查询（Name=域名，Value=权威服务器主机名）
  - **CNAME**：别名→规范主机名（Name=别名，Value=规范名）
  - **MX**：邮件服务器别名映射（Name=域名，Value=邮件服务器规范名）

#### DNS报文
- 查询与回答报文格式相同
- **报文首部（12字节）**：
  - 标识符/标志（16 bit）：匹配请求与响应
  - 标志位（1 bit）：xx数，含查询/应答、权威应答、希望递归、递归可用等标志
  - 计数字段（2 bytes）：问题数、回答RR数、权威RR数、附加RR数（各2字节）
- **报文其余部分（可变长度）**：
  - **问题区域**：含查询的主机名和类型（如A、MX）
  - **回答区域**：返回资源记录（含Type、Value、TTL）
  - **权威区域**：其他权威服务器的记录
  - **附加信息区域**：补充记录（如邮件服务器对应的A记录）

#### DNS记录插入（搭建网站流程）
- **核心步骤**：
  1. 在上级域DNS中插入两条记录：NS记录（指向子域）和A记录（指定权威服务器IP）
  2. 在新增子域的名字服务器上，运行域名服务器DNS Server，负责本域的名字解析：名字 -> IP地址
- **补充**
  - 注册商负责域名注册和记录注入
  - 传统记录静态配置，支持动态更新（UPDATE选项）
  - 完成后全球可通过层级解析访问服务

#### DNS安全
- **DDoS攻击**：
  - 根服务器：流量过滤防护，缓存分流有效
  - TLD服务器：缓存降低实际影响
  - 广播放大：放大攻击流量
- **欺骗攻击**：
  - 拦截DNS查询：返回虚假回复，重定向用户
  - 缓存中毒：诱使服务器缓存伪造结果
- **防护**：DNSSEC（RFC 4033）作为安全版本，防范中间人与缓存投毒；实际攻击中过滤和缓存可减轻影响


## 2.5 P2P应用

### 2.5.1 P2P文件分发

### 2.5.1纯P2P架构

- 没有（或极少）一直运行的服务器
    
- 任意端系统都可以直接通信
    
- 节点（Peers）向其他节点请求服务，并向其他节点提供服务
    
    - 自扩展性：新的节点带来新的业务容量和新的业务需求
        
- 节点间歇上网，每次IP地址都有可能变化
    
    - 管理复杂

#### 共有数据和问题

- 探讨从一台服务器分发大小为$F$的文件到$N$个节点的时间
    
- $u_s$：服务器上载速率
    
- $d_{min}$：所有客户端中最小下载速率
    

#### C/S模式

- **服务器端**：需为$N$个客户端各传输一份文件拷贝，总传输量为$NF$，总耗时为$\frac{NF}{u_s}$
    
- **客户端**：每个客户端需下载完整文件，所有客户端完成下载的最小耗时至少为$\frac{F}{d_{min}}$
    
- **C/S最小分发时间**：$D_{C-S} = \max\left\{\frac{NF}{u_s},\ \frac{F}{d_{min}}\right\}$
    
- **※**分发时间随节点数量$N$的增加呈线性增长
    

#### P2P模式

- **服务器端**：仅需上载一份文件拷贝，耗时为$\frac{F}{u_s}$
    
- **客户端**
    
    - 每个客户端下载一个拷贝，最小耗时为$\frac{F}{d_{min}}$
        
    - 所有客户端总体下载量为$NF$，最大上载总带宽为$u_{total}=u_s+\sum u_i$，系统完成分发的总耗时为$\frac{NF}{u_s+\sum u_i}$
        
- **P2P最小分发时间**：$D_{P2P} = \max\left\{\frac{F}{u_s},\ \frac{F}{d_{min}},\ \frac{NF}{u_s+\sum_{i=1}^{N} u_i}\right\}$
  - 分发时间随节点数量$N$增加呈**非线性增长**，节点可同时作为下载者与上传者，具备自扩展性
    

#### 核心对比与实例

- 设定实例：客户端上载速率$u=u_i$，$\frac{F}{u}=1$小时，服务器上载速率$u_s=10u$，$d_{min}\ge u_s$
    
- **C/S模式**：节点数量$N$增加时，分发时间呈线性快速增长，无上限
    
- **P2P模式**：最小分发时间始终小于服务器单独分发时间（$\frac{F}{u_s}$），且$N$足够大时分发时间不超过1小时，具备显著的扩展性优势
    

#### 结论

- C/S模式分发效率受服务器带宽限制，节点越多耗时越长
    
- P2P模式利用节点的上传能力分担服务器负载，节点数量越多分发效率越高，是具备**自扩展性**的文件分发方案
### 2.5.3 BitTorrent：基于文件分发的P2P协议

#### 核心术语
- **Torrent（洪流）**：参与特定文件分发的所有对等方集合
- **chunk（块）**：文件划分的等长块（典型256KB）
- **tracker（追踪器）**：跟踪洪流中对等方的基础设施节点
- **churn（扰动）**：节点随时上线或下线的特性

#### 节点加入洪流流程
1. 初始无任何块，通过其他节点累积
2. 向tracker注册并周期性通知在线状态
3. tracker返回对等方子集，节点建立TCP连接形成邻近对等方
4. 邻近对等方随时间动态变化

#### 文件块请求机制
- 节点周期性询问邻居持有的块列表
- **最稀缺优先（rarest first）**：优先请求邻居中副本最少的块，均衡副本分布，加速分发

#### 文件块发送机制：一报还一报（tit-for-tat）
- **疏通（unchoked）**：每10秒向提供带宽最高的4个对等方发送块
- **阻塞（choked）**：其余邻居无法获取数据
- **优化疏通**：每30秒随机选择1个额外邻居发送块，使新节点有机会成为高带宽对等方，实现动态优化

#### 补充
- 激励机制为一报还一报，广泛成功
- DHT（分布式散列表）在BitTorrent中广泛实现，用于分布式记录存储


### 2.6 视频分发和内容分发网络

### 2.6.1 因特网视频的基本概念
- **核心挑战**：规模性（服务巨量用户）和异构性（不同带宽接入能力）
- **解决方案**：分布式应用层基础设施（CDN）
- **视频基础**：图像序列，编码利用空间冗余（同帧内）和时间冗余（帧间）压缩比特率
- **流式存储视频约束**：需连续播放，通过客户端缓冲区应对网络抖动，支持交互操作（快进/跳转）

### 2.6.2 流式多媒体技术DASH
- **定义**：基于HTTP的动态自适应流式，客户端根据网络状况选择视频版本
- **核心流程**：
  - 获取告示文件（含各版本URL及比特率）
  - 通过HTTP GET请求字节范围，动态切换不同码率视频块
- **客户端决策**：根据缓存状态决定请求时机，根据带宽选择码率，根据服务器状态选择来源
- **公式**：流式视频 = 编码 + DASH + 播放缓冲区

### 2.6.3 内容分发网（CDN）
- **定义**：分布式多位置服务器，存储内容副本，定向用户至最优节点
  - 有规模性和异构性
- **部署方案**：
  - **深入式（enter deep）**：节点靠近接入网，贴近用户（如Akamai），管理难度高
  - **邀请做客（bring home）**：部署于关键IXP节点，维护开销低，但时延略高
- **工作逻辑**：采用拉策略，未存储内容时从源站或其他集群获取并缓存
- **访问流程**：DNS解析返回CNAME，定向至CDN权威DNS，返回最优集群IP，客户端请求内容（DASH返回告示文件）
- **集群选择策略**：
  - 地理邻近：基于LDNS IP定位，可能不准确
  - 实时测量：周期性探测时延/丢包，部分LDNS不响应
- **分类**：专用CDN（自有内容）和第三方CDN（代理多内容提供商）

### 2.6.4 实际案例
- **Netflix**：云存储多版本视频，CDN分发，客户端通过DASH自适应获取
- **Bilibili**：多层CDN架构（多云调度 + 自建/P2P边缘节点补充）
